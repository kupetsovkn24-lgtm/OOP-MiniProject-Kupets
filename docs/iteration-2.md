# Ітерація 2 - Lab 35

## Що готово

- Повний сценарій ремонту: створення запису, підтвердження, створення роботи, призначення механіка, старт і завершення з розрахунком ціни.
- Сценарій скасування запису до початку ремонтної роботи.
- Асинхронні `SaveAllAsync` і `LoadAllAsync` для JSON-файлів.
- Відновлення стану при запуску програми і через пункт меню.
- Запити: записи на дату, роботи механіка за період, завершені роботи, виручка і кількість робіт по механіках.

## Бізнес-правила

| Правило | Де реалізовано | Де перевірено |
| --- | --- | --- |
| Запис не можна створити у минулому | `Appointment`, `CreateAppointmentUseCase` | `AppointmentTests` |
| Зайнятий слот авто не можна забронювати вдруге | `CreateAppointmentUseCase`, `IAppointmentRepository` | `UseCaseTests` |
| `Job` створюється лише з підтвердженого запису | `JobFactory` | `JobFactoryTests` |
| Зайнятого механіка не можна призначити на конфліктну роботу | `AssignMechanicUseCase` | `UseCaseTests` |
| Завершену роботу не можна скасувати або повторно змінювати | `Job` | `JobStatusTests`, `UseCaseTests` |
| Запис не можна скасувати, якщо для нього вже створено роботу | `CancelAppointmentUseCase` | `UseCaseTests` |

## Зміни класів і контрактів

- `JsonPersistence` працює асинхронно і повідомляє про пошкоджені або конфліктні JSON-дані.
- `InMemoryRepository<T>` отримав `ReplaceAll`, щоб повторне завантаження замінювало стан без дублікатів.
- Додано `CancelAppointmentUseCase` і `CancelJobUseCase`.
- `WorkshopFacade` відкриває скасування і запит завершених робіт для Console.

## Патерн розширення

Основний патерн для вимоги лабораторної - `Strategy`: `IPricingStrategy` дозволяє вибрати розрахунок ціни без зміни `Job` або Console. Без нього правила ціни довелося б прописувати умовами в одному сервісі. На Lab 37 можна додати стратегію знижки або термінового ремонту окремим класом.

Також у проєкті вже використовуються `Factory`, `Observer` і `Facade`, але для Lab 35 не створювалися зайві нові патерни.

## Persistence і помилки

- Відсутні JSON-файли означають порожній початковий стан.
- Пошкоджений JSON завершує завантаження контрольованою помилкою `InvalidDataException`.
- Некоректні зв'язки між сутностями також відхиляються без часткового завантаження.
- Дані замінюються в репозиторіях тільки після успішного читання всього набору файлів.

# Scope практики

## В scope

Цикл життя одного запису від дзвінка до закритої роботи:

1. **Створення Appointment** з валідацією часу
2. **Створення Job** на основі Appointment (через Factory)
3. **Призначення механіка** з перевіркою зайнятості
4. **Переходи статусів** Job: Created → Assigned → InProgress → Completed
5. **Розрахунок вартості** через Strategy (Labor + Parts)
6. **Збереження/відновлення** стану через JSON

## Сутності в scope

- `Customer` — клієнт СТО
- `Vehicle` — авто (належить Customer)
- `Appointment` — запис на час
- `Job` — фактична робота
- `Mechanic` — виконавець
- `Money` — value object для грошей
- `Vin`, `PhoneNumber` — value objects для валідації

## Бізнес-правила в scope

- Час Appointment не може перетинатися з іншими активними Appointments
- Не можна призначити механіка, у якого вже є активний Job у цьому слоті
- Job не можна завершити, якщо вона не в InProgress
- Job не можна скасувати після InProgress (рішення: спрощення)
- `Money.Amount >= 0`
- Vehicle не існує без Customer
- Job не існує без Appointment
- Завершена Job має фінальну вартість (immutable після Completed)

## Поза scope (і чому)

### Не робимо CRM Саші
**Чому:** окремий продукт, який вже працює. Інтеграція потребує API, доступ до якого ми не маємо. Поза навчальним scope.

### Не робимо склад запчастин
**Чому:** окремий bounded context (inventory management). У моделі залишається тільки **факт** — "на роботу пішли запчастини на суму X". Звідки взялися — не питання.

### Не робимо UI / Web / Laravel
**Чому:** практика про ООП-ядро, не про fullstack. ConsoleApp як демонстрація достатньо.

### Не робимо авторизацію / користувачів / ролі
**Чому:** немає вимоги, додає складності без академічної цінності.

### Не робимо Telegram / повідомлення
**Чому:** інтеграція з зовнішніми API не вимагається.

### Не робимо SaaS / multi-tenant / billing
**Чому:** передчасна абстракція. Немає навіть першого підтвердженого клієнта.

### Не робимо аналітику / звіти / дашборди
**Чому:** базові LINQ-запити в use cases закривають академічну вимогу. Окремий звітний модуль не потрібен.

### Не робимо БД (SQL Server, PostgreSQL)
**Чому:** JSON через DTO покриває вимогу персистентності. БД додає складності без академічної цінності для цієї практики.

### Не робимо State pattern
**Чому:** свідоме рішення. 5 статусів і прості переходи краще читаються через метод з таблицею. State був би виправданий при більшій кількості станів і складніших правилах.

### Не робимо Singleton, Composite
**Чому:** Singleton — антипатерн у сучасному C# (краще DI). Composite — натягнуто на цей домен.

### Не моделюємо AppointmentStatus.NoShow
**Чому:** ускладнює state machine без академічної цінності. Можна додати при розширенні.

## Що залишиться корисним після практики

**Залишається:**
- Domain model і список бізнес-правил
- UML як комунікаційний інструмент з Дімою
- Розуміння домену СТО, перевірене кодом і тестами
- Список відкритих питань для discovery з клієнтом

**Не залишається:**
- Сам C# код (бо потенційний продукт буде на іншому стеку)
- Структура солюшну
- Конкретні технічні рішення (DTO, Mapper, repositories)

Це нормально. Не намагатися це врятувати — гарантований шлях до поганого коду.

# Performance Note

## Перевірений сценарій

Перевірено логіку переходів статусу `Job`, бо вона виконується при кожній основній операції ремонту: призначенні механіка, старті, завершенні або скасуванні.

## Використана структура

У `Job` дозволені переходи описані як:

```text
Dictionary<JobStatus, HashSet<JobStatus>>
```

- `Dictionary` швидко знаходить список дозволених переходів для поточного статусу.
- `HashSet` швидко перевіряє, чи дозволено конкретний наступний статус.
- Код читається як явна таблиця бізнес-правил, а не як довгий набір `if`.

## Мікроаналіз

| Операція | Підхід | Складність |
| --- | --- | --- |
| Знайти правила поточного статусу | `Dictionary` lookup | у середньому `O(1)` |
| Перевірити наступний статус | `HashSet.Contains` | у середньому `O(1)` |
| Зібрати виручку/роботи | LINQ по in-memory списку | `O(n)` |

## Висновок

Для навчального застосунку і невеликого набору даних цього достатньо. Перенесення статистики у складні індекси або БД зараз ускладнило б проєкт без помітної користі. Якщо даних стане багато, першою реальною оптимізацією має бути заміна файлового persistence на сховище із запитами та індексами.

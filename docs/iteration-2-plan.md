# План ітерації 2 - Lab 35

## Сценарії в роботі

- Повний цикл ремонту: запис, підтвердження, робота, механік, завершення і вартість.
- Скасування запису до створення ремонтної роботи.
- Збереження й відновлення стану СТО з JSON.
- Перегляд робіт і короткої статистики для адміністратора.

## Що не змінюємо

- Основні сутності `Customer`, `Vehicle`, `Appointment`, `Job`, `Mechanic`.
- Value objects `Money`, `Vin`, `PhoneNumber`.
- Наявні `Strategy`, `Factory`, `Observer` і `Facade`, бо вони вже відповідають предметній області.

## Точки розширення

- `IPricingStrategy` для різних способів розрахунку вартості.
- Repository interfaces для роботи application-рівня без прив'язки до JSON.
- `WorkshopFacade` як проста точка входу для меню.
- DTO та mapper для безпечного відновлення доменних сутностей із файлів.

## Ризики

- Пошкоджений JSON або посилання на відсутню сутність можуть зламати відновлення стану.
- Скасування запису після створення `Job` створило б суперечливі дані.
- Бізнес-правила не можна дублювати в Console, інакше їх буде важко тестувати.

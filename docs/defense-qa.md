# Defense Questions and Answers

## 1. Чому є окремі шари?

`Domain` містить правила, `Application` керує сценаріями, `Infrastructure` працює з JSON, а `Console` лише взаємодіє з користувачем. Так правила простіше тестувати й змінювати.

## 2. Де тут справжня бізнес-логіка?

У перевірках часу запису, конфлікту механіка, дозволених статусів `Job`, забороні дублювання роботи та правилах скасування.

## 3. Навіщо Strategy?

Різні формули ціни реалізуються окремими класами через `IPricingStrategy`. `Job` не потрібно змінювати при появі нового тарифу.

## 4. Чому JSON, а не база даних?

Для консольного навчального проєкту JSON достатньо, щоб показати serialization, async I/O і fault handling. База даних збільшила б scope без необхідності для лабораторної.

## 5. Як обробляється зламаний файл?

`JsonPersistence` викидає `InvalidDataException`, записує діагностичне повідомлення і не замінює репозиторії частково відновленими даними.

## 6. Чим unit tests відрізняються від integration tests у проєкті?

Unit tests перевіряють правила й use cases із контрольованими залежностями. Integration tests перевіряють повний цикл із реальними JSON-файлами у тимчасовій папці.

## 7. Чому coverage не означає ідеальну якість?

Coverage показує виконані рядки, але не доводить правильність перевірок. Тому є окремі тести негативних сценаріїв і матриця use cases.

## 8. Який рефакторинг зроблено перед релізом?

З `WorkshopFacade` прибрано невикористаний repository parameter і додано коротку документацію ключових API. Це невелика зміна, яка робить залежності чеснішими.

## 9. Що залишилось поза релізом?

База даних, GUI, авторизація, склад запчастин і транзакційне збереження. Вони не потрібні для навчальної мети `v1.0.0`.

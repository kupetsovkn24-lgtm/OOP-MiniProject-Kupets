# Бриф для VS Code Claude

> Цей файл — точка входу для Claude у VS Code. Прочитавши його, він має зрозуміти контекст роботи без додаткових пояснень.

## Хто я і що роблю

Я студент, що проходить навчальну практику з ООП. Тема: **Система управління ремонтними задачами СТО**. Стек: **C# / .NET**.

## Що це за проєкт

Це **навчальне ООП-ядро**, не бойовий продукт. Мета — здати практику + отримати валідну доменну модель, яка може стати mental model для майбутнього потенційного клієнта (СТО Діми).

## Перед тим як щось робити — прочитай

1. `docs/PLAN.md` — план по тижнях
2. `docs/SCOPE.md` — що в scope, що поза ним
3. `docs/DOMAIN.md` — опис домену і бізнес-правил
4. `docs/DECISIONS.md` — архітектурні рішення з обґрунтуванням

Якщо щось у цих документах суперечить моїй поточній інструкції — спитай, не вгадуй.

## Принципи роботи

**Чого я очікую:**
- мінімалізм у межах академічних вимог (патерни — там, де виправдано, але обов'язкові з маршруту мають бути)
- чистий C# без фреймворкової магії
- Domain не залежить від Infrastructure
- бізнес-логіка покрита тестами
- кожне неочевидне рішення — з коментарем "чому"

**Чого я НЕ очікую:**
- бойових production-ready патернів (DI-контейнери, mediator, CQRS)
- БД, ORM, Web API, UI
- "як прийнято в індустрії" — у нас академічний контекст

## Поточний етап

Перед початком кодування дивись `docs/PLAN.md` — там видно, на якому ми тижні.

**Якщо я кажу "робимо тиждень N"** — починай з критеріїв готовності цього тижня в плані, не пиши код раніше за моделювання.

- пояснюй причинно-наслідково

- якщо є простіший варіант — запропонуй
- якщо щось нечітко — питай, не вгадуй

Стратегічні і архітектурні рішення верхнього рівня обговорюються в окремому чаті (web Claude). Якщо я раптом приношу нові вимоги "звідкись" — це звідти. Не сперечайся з ними, але можеш ставити уточнюючі питання якщо є технічні протиріччя.

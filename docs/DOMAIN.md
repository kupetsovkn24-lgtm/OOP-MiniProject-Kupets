# Опис домену: СТО

## Реальний контекст (з візиту)

СТО Діми. Робочий процес гібридний: частина паперова, частина в CRM.

### Учасники

- **Діма** — власник. Хоче прозорість по операціям і фінансам.
- **Ваня** — адмін, колишній майстер. Веде записи в зошиті, приймає дзвінки, координує машини між майстрами.
- **Саша** — менеджерка. Сидить за ПК, користується CRM по підписці ($100/міс). Веде клієнтів, замовляє запчастини, базові фінанси.
- **Майстри** — спеціалізовані: ходова, розвал, зварювання, двигуни.

### Інформаційні світи

1. **Зошит Вані** — операційне розкладання дня. Хто, коли, до якого майстра. Невидимий ні для кого крім нього.
2. **CRM Саші** — клієнти, запчастини, частина фінансів.
3. **Excel Діми** — спроба бачити фінанси.

**Ключова проблема:** зошит Вані — серце операційки, і ніхто крім Вані його не бачить.

---

## Реальний цикл клієнта

1. Дзвінок → Ваня або Саша приймає
2. Запис на годину → ручкою в зошит
3. Призначення майстра → за спеціалізацією
4. Саша замовляє запчастини під запис
5. День ремонту → авто приїжджає
6. Майстер виконує → фотографує авто з номером
7. Список виконаних робіт
8. Оплата

## Доменна модель (для практики)

### Сутності

| Сутність | Тип | Опис |
|----------|-----|------|
| `Customer` | Entity | Клієнт СТО |
| `Vehicle` | Entity | Авто, належить Customer |
| `Appointment` | Aggregate Root | Запис на час |
| `Job` | Aggregate Root | Фактична робота |
| `Mechanic` | Entity | Виконавець з спеціалізацією |
| `Money` | Value Object | Грошова сума |
| `Vin` | Value Object | VIN з валідацією |
| `PhoneNumber` | Value Object | Телефон з валідацією |

---

### Зв'язки

- `Customer 1 — 0..* Vehicle`
- `Vehicle 1 — 0..* Appointment`
- `Appointment 1 — 0..1 Job`
- `Job 1 — 1 Mechanic`
- `Mechanic 1 — 0..* Job`

**Customer на Appointment не зберігається окремо**, а дістається через `Vehicle.Owner`.

---

## Стани Job

```

Created → Assigned → InProgress → Completed
↓         ↓
Cancelled  Cancelled

```

- Job створюється у стані **Created**
- Механік призначається окремим кроком

**Свідомо НЕ моделюємо:**
- `InProgress → Cancelled` (спрощення для практики)

---

## Стани Appointment

```

Created → Confirmed → (стає Job)
↓          ↓
Cancelled  Cancelled

```

**Свідомо НЕ моделюємо:**
- `NoShow` (зайве ускладнення)

## Бізнес-правила (інваріанти)

### Money
- `Amount >= 0`
- `Currency` immutable
- Операції тільки між однаковими валютами
- Equality by value

---

### Customer
- `FullName` не порожній
- `Phone` валідний через `PhoneNumber`

---

### Vehicle
- `Vin` валідний
- `Owner` обов'язковий
- `Brand`, `Model` не порожні

---

### Appointment
- `ScheduledAt` у майбутньому
- Не перетинається з іншим Appointment цього Vehicle
- `ProblemDescription` не порожній

---

### Job

- Створюється тільки через `JobFactory.CreateFromAppointment(...)`
- Створюється у статусі `Created`
- Не має механіка при створенні

#### Призначення механіка
- Виконується через `AssignMechanicUseCase`
- Після призначення статус → `Assigned`
- Дозволено тільки якщо механік вільний

#### Життєвий цикл
- `Assigned → InProgress` — тільки якщо механік доступний
- `InProgress → Completed` — тільки з валідною вартістю

#### Вартість
- Складається з:
  - `LaborCost`
  - `PartsCost`
- Розрахунок через `IPricingStrategy`

#### Після завершення
- Job стає immutable
- Вартість більше не змінюється

---

### Mechanic
- Має `Specialization` (enum)
- Зайнятість не зберігається в самому класі
- Перевіряється через repository
- Не зберігає список активних Job

---

## Edge cases (поза scope)

| Ситуація | Рішення |
|----------|---------|
| Клієнт не приїхав | не моделюється |
| Запчастина не прийшла | поза scope |
| Механік захворів | не моделюється |
| Скасування після InProgress | не дозволено |
| Несплачений Job | не моделюється |
| Кілька авто | підтримується |
| Один механік на кілька авто | перевіряється через слоти |

---

## Словник домену

- **Appointment** — запис (зошит Вані)
- **Job** — фактична робота
- **Labor cost** — оплата праці
- **Parts cost** — вартість запчастин
- **Slot** — часовий проміжок
- **Pricing strategy** — спосіб розрахунку вартості

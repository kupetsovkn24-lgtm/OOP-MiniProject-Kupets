# Test Strategy - Lab 36

## Критичні сценарії

- Повний ремонт: запис -> підтвердження -> робота -> механік -> завершення -> ціна.
- Збереження стану у JSON і коректне відновлення після нового запуску.
- Скасування запису без появи суперечливих даних.
- Заборона неправильних переходів статусів і конфліктів механіка/часу.

## Що перевіряється unit tests

- Інваріанти `Appointment`, `Job`, `Money` та інших domain-об'єктів.
- Бізнес-правила use cases із mock-репозиторіями.
- `IPricingStrategy`, factory, observer та LINQ-запити.

Unit tests доречні тут, бо правила мають працювати без файлової системи й Console.

## Що перевіряється integration tests

- Взаємодія `WorkshopFacade`, реальних in-memory repositories і `JsonPersistence`.
- Save/load повного стану через реальні тимчасові JSON-файли.
- Поведінка після відновлення: нова операція, статистика та повторне збереження.

## Негативні сценарії

- Пошкоджений JSON.
- Посилання DTO на відсутню сутність.
- Відсутні файли при першому запуску.
- Спроба скасувати запис після створення роботи.
- Спроба змінити завершену роботу.

## Точковий рефакторинг

- У `JsonPersistence` додано необов'язковий callback журналювання помилок.
- Побічний ефект логування тепер можна перевірити в тесті без Console.
- Для integration tests використовується локальна fixture-обгортка над тимчасовою директорією.

Стратегія відмови проста: для пошкоджених даних завантаження завершується контрольованою помилкою без часткового перезапису стану; автоматичний retry не додається, бо пошкоджений JSON повторною спробою не виправляється.

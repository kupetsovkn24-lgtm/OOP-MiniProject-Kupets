# Vision

## Назва проєкту

CarWorkshop - система управління ремонтними задачами СТО.

## Проблема

Невелике СТО веде записи клієнтів, ремонтні роботи та призначення майстрів вручну або в розрізнених інструментах. Через це легко отримати зайнятий часовий слот, призначити недоступного механіка або втратити зв'язок між записом, фактичною роботою і фінальною вартістю.

Проєкт моделює навчальне ООП-ядро, яке показує повний цикл: запис клієнта -> створення роботи -> призначення механіка -> зміна статусів -> завершення -> розрахунок вартості.

## Цільові користувачі

- Адміністратор СТО: створює записи, підтверджує візит, призначає механіка.
- Механік: виконує роботу, яка проходить життєвий цикл статусів.
- Власник або менеджер СТО: бачить завершені роботи й підсумкову вартість.

## Основні сценарії використання

1. Створення запису клієнта на ремонт із перевіркою вільного часу.
2. Створення ремонтної роботи на основі підтвердженого запису.
3. Призначення вільного механіка на роботу.
4. Запуск і завершення роботи з розрахунком вартості.
5. Збереження та відновлення стану через JSON-файли.
6. Скасування запису, якщо ремонтна робота ще не була створена.

## Нефункціональні вимоги

- Код має бути читабельним і розділеним на шари `Domain`, `Application`, `Infrastructure`, `Console`.
- Бізнес-правила мають бути тестованими без запуску UI або бази даних.
- Домен має бути розширюваним: нові стратегії ціноутворення або способи збереження не повинні вимагати переписування сутностей.
- Помилки предметної області мають бути явними через доменні винятки.
- Проєкт має збиратися й тестуватися автоматично через CI.

## Обмеження першої ітерації

На Lab 34 свідомо не реалізуються:

- повноцінний CRM;
- веб-інтерфейс або GUI;
- авторизація та ролі користувачів;
- база даних;
- склад запчастин;
- Telegram або інші інтеграції;
- SaaS/multi-tenant логіка;
- складна аналітика.

Причина: мета першої ітерації - стабільний архітектурний каркас, доменна модель, вертикальний зріз і артефакти для наступних лабораторних.

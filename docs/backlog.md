# Backlog

## Ітерація 1 - Lab 34: baseline

Мета: створити основу проєкту, яка буде розвиватися в наступних лабораторних.

Пріоритети:

- Зафіксувати vision, scope, backlog і UML-артефакти.
- Створити solution з шарами `Domain`, `Application`, `Infrastructure`, `Console`, `Tests`.
- Реалізувати предметну модель СТО: `Customer`, `Vehicle`, `Appointment`, `Job`, `Mechanic`.
- Додати value objects: `Money`, `Vin`, `PhoneNumber`.
- Реалізувати repository interfaces та in-memory repositories.
- Реалізувати перший вертикальний зріз через консоль.
- Додати базові unit-тести.
- Налаштувати README, `.gitignore` і GitHub Actions CI.

## Ітерація 2 - Lab 35: бізнес-логіка, persistence, запити

Мета: розширити поведінку системи без зміни архітектурного каркаса.

План:

- Розширити сценарії роботи із записами: cancelled/no-show або перенесення часу.
- Додати запити по роботах механіка за період.
- Покращити JSON persistence і обробку помилок читання/запису.
- Додати use cases для перегляду активних робіт і завершених робіт.
- Уточнити межі між доменними інваріантами та application-сервісами.

## Ітерація 3 - Lab 36: quality gate, тести, fault handling

Мета: посилити якість, стабільність і регресійний захист.

План:

- Розширити unit-тести для edge cases.
- Додати тести для помилкових сценаріїв persistence.
- Перевірити code smells і прибрати зайве дублювання.
- Уточнити стратегію винятків і Result-подібних відповідей там, де це доречно.
- Додати quality gate у CI, якщо це буде вимогою лабораторної.

## Ітерація 4 - Lab 37: release, документація, демо

Мета: підготувати фінальну демонстрацію та документацію.

План:

- Оновити README під фінальний стан.
- Додати короткий demo script для захисту.
- Перевірити UML на відповідність реальному коду.
- Підготувати release notes/changelog.
- Перевірити, що репозиторій не містить generated-файлів.

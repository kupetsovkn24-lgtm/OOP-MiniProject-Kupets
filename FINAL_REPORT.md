# Final Report - CarWorkshop v1.0.0

## Проєкт

`CarWorkshop` - навчальний міні-проєкт на C#/.NET для керування ремонтною задачею невеликого СТО. Його мета не замінити реальну CRM, а показати завершений ООП-сценарій з бізнес-правилами, збереженням і тестуванням.

## Реалізований результат

- Повний сценарій: запис -> підтвердження -> робота -> механік -> виконання -> фінальна вартість.
- Скасування запису й роботи у дозволених станах.
- JSON save/load із DTO і контрольованою обробкою пошкоджених даних.
- Запити для перегляду завершених робіт, робіт механіка та виручки.
- Звіт по дорогих завершених ремонтах із гнучким delegate-фільтром.
- CI, unit tests, integration tests і coverage report.

## Архітектура

Рішення поділено на `Domain`, `Application`, `Infrastructure`, `Console` і `Tests`. Домен не знає про JSON або Console. Application координує сценарії через інтерфейси репозиторіїв. Infrastructure містить конкретне файлове збереження.

Використані патерни:

| Патерн | Для чого використано |
| --- | --- |
| Repository | Відокремити бізнес-логіку від способу збереження |
| Strategy | Різні способи розрахунку вартості |
| Factory | Створення `Job` тільки з коректного `Appointment` |
| Observer | Історія статусів роботи |
| Facade | Простий API для консольного меню |
| Decorator | Логування викликів repository |

## Фінальний рефакторинг

- З конструктора `WorkshopFacade` прибрано невикористаний `ICustomerRepository`. Це прибрало хибну залежність і зробило API точнішим.
- Додано XML-коментарі для `WorkshopFacade`, `IPricingStrategy` та `JsonPersistence`, бо саме ці типи є головними точками розширення і роботи з файлами.
- Додаткове розбиття класів не робилося: після Lab 36 великого дублювання або перевантаженого класу, який заважає демо, не виявлено.
- Нові фічі перед релізом не додавалися: стабільність і пояснюваність важливіші за зайвий scope.

## Дані та продуктивність

Переходи статусу `Job` перевіряються через `Dictionary` і `HashSet`, що дає очікуваний пошук `O(1)` і залишає правила читабельними. LINQ-запити працюють за `O(n)` по in-memory наборах, що доречно для навчального малого застосунку. Деталі: [docs/performance-note.md](docs/performance-note.md).

## Розширення IW29

- `FilterCompletedJobs(...)` приймає `Func<Job, bool>` і дозволяє задавати умову звіту лямбда-виразом.
- `CompletedJobsReport` збирає відібрані завершені роботи, їх кількість і загальну виручку.
- Console демонструє звіт ремонтів від `4000 UAH`, а unit tests перевіряють дорогі, дешеві, незавершені та порожні вибірки.

Це три залежні кроки: гнучкий фільтр став основою звіту, а звіт став основою тестування і демонстрації. Деталі зафіксовано у [docs/extension-report.md](docs/extension-report.md).

## Компроміси й ризики

- JSON підходить для курсового демо, але не гарантує транзакційність кількох файлів.
- Немає бази даних, резервного копіювання або авторизації.
- Ці обмеження свідомо залишено поза `v1.0.0`, бо вони не потрібні для вимог лабораторних.
